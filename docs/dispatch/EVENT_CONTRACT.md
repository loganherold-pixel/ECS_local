# ECS Dispatch Event Contract

This document defines the product-level Dispatch events emitted, persisted, synced, logged, or notified by the ECS Dispatch feature.

It is a documentation-only contract. It describes the current adapter-based implementation and the intended interpretation of Dispatch event names without changing app behavior.

## Event Channels

Dispatch events can appear in several channels:

- **Local Dispatch state and persistence**: pings, queue items, assignments, assist requests, acknowledgments, and timeline events are staged through Dispatch adapters and local persistence.
- **Timeline**: user-facing operational history backed by `DispatchTimelineEvent.type` values such as `ping_created`, `queue_escalated`, `queue_resolved`, `assignment_created`, `assist_request_created`, `sync`, and `log`.
- **Expedition log**: privacy-safe audit/log staging is gated by Dispatch rollout configuration. When disabled, timeline remains local to Dispatch.
- **Realtime sync**: active-expedition updates are published through realtime envelope types such as `ping_upsert`, `queue_item_upsert`, `assignment_upsert`, `team_member_upsert`, and `timeline_event_added`.
- **Notifications**: notification policy is isolated in the Dispatch notification adapter. Notifications are rollout-gated and deduped; emergency and assist copy must remain ECS team coordination only.
- **Audit mapping**: timeline events can be converted into privacy-safe audit events such as `team_ping_created`, `queue_item_escalated`, `assist_request_created`, `emergency_assist_request_created`, `offline_action_replayed`, `failed_delivery`, and `permission_denied_attempt`.

Some event names below are **product-level events**, not literal `DispatchTimelineEvent.type` strings. For example, `ping_sent`, `ping_queued`, `ping_delivered`, `ping_recovered`, and `ping_failed` are represented primarily as ping delivery/status transitions and may only create timeline/audit events when the action is operationally meaningful.

## Shared Event Fields

Every Dispatch event should be scoped and attributable when it is persisted or logged:

- **Actor**: current ECS user, system adapter, or local device fallback.
- **Expedition scope**: active expedition ID when available.
- **Target**: recipient member, role/group, all team, queue item, assignment, linked context, or notification recipient set.
- **Related IDs**: stable entity IDs for the ping, queue item, assignment, assist request, acknowledgment, timeline event, notification policy key, or offline action.
- **Linked context**: optional `DispatchLinkedContext` for expedition, pin, waypoint, route segment, resource, vehicle, power, or manual context.
- **Idempotency**: stable IDs and idempotency keys are used where supported to prevent duplicate pings, queue items, timeline events, offline replay writes, realtime echoes, and notification sends.

## Duplicate Prevention

Dispatch duplicate prevention is layered:

- Stable IDs are used for pings, queue items, assignments, assist requests, acknowledgments, timeline events, and queued local actions.
- Idempotency keys are created for related action bundles so a ping, queue item, and timeline event created together can be replayed without duplicating.
- Rapid local submits are guarded before persistence.
- Incoming realtime updates are merged by stable ID and update metadata instead of appended blindly.
- Timeline/log events are deduped by stable timeline ID and audit-safe related entity references.
- Notification policy uses deterministic notification keys and a cooldown window to avoid spam from optimistic updates, realtime echoes, retries, and offline replay.

## Privacy And Safety

Dispatch events must remain scoped to the active expedition and must respect permission adapters before action execution or private rendering.

- Member location and contact details are restricted unless ECS permissions allow access.
- Assist Request and Emergency Ping are ECS team coordination only.
- Dispatch does not contact emergency services, SMS, email, or phone services unless ECS has verified infrastructure and a future task explicitly enables it.
- Audit/log descriptions should avoid raw technical IDs, sensitive coordinates, and private contact details.

## Known Limitations

- Several product-level events are currently represented as status transitions rather than literal timeline event type strings.
- Expedition log integration is rollout-gated and may remain disabled while Dispatch keeps local timeline history.
- Notification behavior is rollout-gated and policy-driven; documentation of `notification_sent` and `notification_suppressed` reflects adapter outcomes, not a separate persisted notification event table.
- Offline replay depends on the existing ECS sync/offline infrastructure exposed through Dispatch adapters; this contract does not introduce a new offline queue backend.
