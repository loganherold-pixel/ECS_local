# ECS Dispatch Data Contract

Last updated: 2026-04-24

## Scope

This document describes the Dispatch data contract implemented in the repository. The source of truth for type definitions is `lib/dispatchTypes.ts`. Persistence, idempotency, realtime, replay, and permission behavior are implemented through the Dispatch adapters and helpers:

- Types: `lib/dispatchTypes.ts`
- ID/idempotency/merge/conflict logic: `lib/dispatchIntegrity.ts`
- Local persistence: `lib/dispatchPersistenceAdapter.ts`
- Realtime transport: `lib/dispatchRealtimeAdapter.ts`
- Offline replay: `lib/dispatchOfflineReplayAdapter.ts`
- Sync/delivery labels: `lib/dispatchSyncAdapter.ts`
- Permissions/privacy: `lib/dispatchPermissionAdapter.ts`
- Notification policy: `lib/dispatchNotificationAdapter.ts`

Dispatch is currently local-first. Dedicated backend Dispatch tables are not present in this repo; `dispatchPersistenceAdapter` stores Dispatch pings, queue items, assignments, and timeline events in a persisted local snapshot.

## Cross-Cutting Contract

### Expedition Scoping

Most live Dispatch state is scoped by active expedition ID. The active expedition context is resolved in `lib/dispatchServiceAdapters.ts`.

Fields and helpers that carry expedition scope:

- `DispatchIdempotencyInput.expeditionId`
- `DispatchRealtimeEnvelope.expeditionId`
- `DispatchPersistenceSnapshot.expeditionId`
- `DispatchAuditEvent.expeditionId`

Realtime sessions subscribe to a channel named by expedition ID: `ecs-dispatch:${expeditionId}`.

### ID Strategy

Local Dispatch entity IDs are generated by `createDispatchEntityId(entityType, idempotencyKey)` in `lib/dispatchIntegrity.ts`.

Prefixes:

- `ping`: `local-ping-*`
- `queue_item`: `local-queue-*`
- `assignment`: `local-assignment-*`
- `assist_request`: `local-assist-*`
- `acknowledgment`: `local-ack-*`
- `timeline_event`: `local-timeline-*`
- `offline_action`: `local-offline-action-*`
- `notification`: `local-notification-*`

IDs are derived from a stable hash of the idempotency key. Realtime envelopes use a separate ID format: `dispatch-rt-${event.type}-${recordKey}`.

### Idempotency Key Strategy

`createDispatchIdempotencyKey` creates a deterministic key from:

- expedition ID
- entity type
- action type
- actor member ID
- sorted target member IDs
- linked context ID
- source entity ID
- normalized message
- priority
- optional time bucket
- optional metadata

Target member IDs are deduplicated and sorted. Messages are normalized to reduce whitespace-based duplicates. This is used to prevent rapid double submissions, offline replay duplicates, realtime echo duplication, and duplicate timeline/log events.

### Duplicate Prevention

Implemented duplicate controls:

- `rememberDispatchAction` blocks rapid duplicate local actions within a default 2.5 second window.
- `mergeDispatchPing`, `mergeDispatchQueueItem`, `mergeDispatchAssignment`, `mergeDispatchAssistRequest`, and `mergeDispatchTimelineEvent` merge records by `id` or `idempotencyKey`.
- `shouldApplyIncomingDispatchEvent` rejects incoming realtime records that are older or do not produce a meaningful merge.
- Realtime sessions remember the last 200 event IDs per session.
- Notification policy has a five-minute dedupe window per notification key.

### Version And Timestamp Behavior

Many Dispatch records include optional `version`. Merge helpers prefer higher `version`, then newer timestamps. Timestamp fields vary by entity:

- `DispatchPing`: `createdAt`, optional `updatedAt`
- `DispatchQueueItem`: `createdAt`, required `updatedAt`
- `DispatchAssignment`: `assignedAt`, optional `updatedAt`
- `DispatchTimelineEvent`: `occurredAt`
- `DispatchAssistRequest`: `createdAt`, optional `updatedAt`
- `DispatchAcknowledgment`: `acknowledgedAt`

Retry/cancel helpers increment `version` where supported and update timestamps where appropriate.

### Actor And Current User Fields

Actor/current user references are stored as member IDs or display labels, not private contact details.

- `DispatchPing.createdByMemberId`
- `DispatchQueueItem.createdByMemberId`
- `DispatchAssignment.assigneeMemberId`
- `DispatchTimelineEvent.actor`
- `DispatchTimelineEvent.target`
- `DispatchAuditEvent.actor.memberId`
- `DispatchAssistRequest.createdByMemberId`
- `DispatchAcknowledgment.memberId`

Permission checks determine whether the current user can view location/contact data or mutate Dispatch records.

### Conflict Fields

Conflict-aware entities can carry:

- `conflictState`: `none`, `updated_during_sync`, `needs_review`
- `conflictReason`
- `lastConflictAt`

These fields are used when offline/realtime updates collide.

## Shared Enums

### DispatchPriority

Values: `low`, `normal`, `high`, `critical`.

Priority affects sorting, escalation recommendations, assist/emergency behavior, notification policy, and visual hierarchy.

### DispatchDeliveryState

Values:

- `draft`
- `queued`
- `sending`
- `sent`
- `delivered`
- `seen`
- `acknowledged`
- `accepted`
- `declined`
- `no_response`
- `escalated`
- `recovered`
- `failed`
- `retrying`
- `cancelled`

Used by pings, queue items, timeline events, and member sync state.

Lifecycle:

- New offline/unknown actions start as `queued`.
- Live actions usually start as `sent`.
- Failed actions can move to `retrying`, then `recovered` or `failed`.
- Queued work can become `cancelled`.
- Response-oriented records can move through `seen`, `acknowledged`, `accepted`, `declined`, `no_response`, or `escalated`.

### DispatchReliabilityState

Values: `live`, `queued`, `sending`, `sent`, `delivered`, `stale`, `offline_risk`, `recovered`, `failed`, `retrying`, `cancelled`, `unknown`.

This is a derived/auxiliary reliability state used for display and offline/replay behavior.

### DispatchEscalationState

Values:

- `none`
- `monitor`
- `follow_up`
- `escalate_to_lead`
- `broadcast_to_team`
- `recommended`
- `escalated`
- `emergency_unresolved`
- `resolved`
- `recovered`

Lifecycle:

- Most records start as `none`, `monitor`, or `recommended`.
- Manual escalation can move to `follow_up`, `escalate_to_lead`, `broadcast_to_team`, `escalated`, or `emergency_unresolved`.
- Resolved/recovered work moves to `resolved` or `recovered`.

Automated escalation is gated off by default in `lib/dispatchRolloutConfig.ts`.

## Entity Contracts

## 1. DispatchTeamMember

Purpose: Represents a person participating in the Expedition Channel roster.

Required fields:

- `id`
- `displayName`
- `callSign`
- `role`: `owner`, `member`, `viewer`
- `status`: `DispatchTeamMemberStatus`
- `lastSeenAt`
- `syncState`: `DispatchDeliveryState`

Optional fields:

- `currentContext`
- `coordinates`
- `batteryPercent`
- `notes`

Status values:

- `connected`
- `offline`
- `on_route`
- `at_waypoint`
- `at_camp`
- `needs_check_in`
- `no_response`
- `unavailable`
- `emergency`

Relationships:

- Referenced by `DispatchPing.targetMemberIds`.
- Referenced by `DispatchQueueItem.assignedMemberIds`.
- Referenced by `DispatchAssignment.assigneeMemberId`.
- Referenced by `DispatchTimelineEvent.memberIds`.

Lifecycle:

- Loaded from the Dispatch team adapter.
- May fall back to deterministic mock members.
- May become a solo current-user member when no live team roster is available.
- Realtime can upsert team member status through `team_member_upsert`.

Persistence source:

- Not currently persisted by `dispatchPersistenceAdapter`.
- Loaded through `defaultDispatchAdapters.teamRoster`.

Realtime behavior:

- `team_member_upsert` can update local roster state.
- Realtime envelope ID is built from member ID, last seen, and status.

Offline behavior:

- Member status drives `offline_risk`, `stale`, and queued delivery presentation.

Privacy considerations:

- `coordinates` and `currentContext` may expose location. UI must check `view_member_location`.
- Contact methods are not part of this entity and remain restricted unless existing ECS permissions allow access.

## 2. DispatchPing

Purpose: Structured Team Ping record for status checks, rally calls, assist requests, route/resource/hazard updates, emergency coordination, and general Dispatch messages.

Required fields:

- `id`
- `type`: `DispatchPingType`
- `priority`: `DispatchPriority`
- `status`: `DispatchPingDeliveryStatus`
- `message`
- `createdAt`
- `createdByMemberId`
- `targetMemberIds`
- `escalationState`

Optional fields:

- `idempotencyKey`
- `version`
- `updatedAt`
- `linkedContext`
- `responseDueAt`
- `acknowledgedByMemberIds`
- `requiresAcknowledgment`
- `checkInType`
- `checkInSchedule`
- `checkInResponses`
- `reliabilityState`
- conflict fields

Enum/status fields:

- `type`: `check_in`, `rally`, `assist`, `route`, `resource`, `hazard`, `emergency`, `general`
- `status`: all `DispatchDeliveryState` values
- `priority`
- `escalationState`

Relationships:

- May create or link a `DispatchQueueItem` through `DispatchQueueItem.sourcePingId`.
- May create `DispatchTimelineEvent.pingId`.
- Targets `DispatchTeamMember.id` values.
- May carry `DispatchLinkedContext`.
- Acknowledgment is represented by `acknowledgedByMemberIds`, `checkInResponses`, and/or `DispatchAcknowledgment`.

Lifecycle:

- Created by Team Ping composer or Assist Request flow.
- Starts as `sent` if deliverable, otherwise `queued`.
- May become `seen`, `acknowledged`, `accepted`, `declined`, `no_response`, `escalated`, `failed`, `retrying`, `recovered`, or `cancelled`.
- Check-in responses can update `acknowledgedByMemberIds` and `checkInResponses`.

Persistence source:

- Stored locally in `DispatchPersistenceSnapshot.pings`.

Realtime behavior:

- Published as `ping_upsert`.
- Incoming pings are merged by ID/idempotency key and version/timestamp.

Offline behavior:

- Local queued/retryable pings with `local-*` IDs are replayed by `replayQueuedDispatchActions`.
- Successful replay updates to recovered/sent behavior through sync helpers.

Privacy considerations:

- Message content should be privacy-safe.
- Linked context may imply location and should be rendered according to permissions.
- Emergency ping is ECS team coordination only; it does not contact emergency services.

## 3. DispatchQueueItem

Purpose: Operational queue item representing active Dispatch work such as response follow-up, assignments, assist requests, hazards, route checks, resource checks, or offline concerns.

Required fields:

- `id`
- `title`
- `detail`
- `status`: `DispatchQueueItemStatus`
- `priority`
- `createdAt`
- `updatedAt`
- `createdByMemberId`
- `assignedMemberIds`
- `linkedContext`
- `escalationState`
- `deliveryState`

Optional fields:

- `idempotencyKey`
- `version`
- `dueAt`
- `tags`
- `sourcePingId`
- `reliabilityState`
- conflict fields

Status values:

- `new`
- `pending_response`
- `assigned`
- `in_progress`
- `blocked`
- `escalated`
- `needs_review`
- `resolved`
- `cancelled`

Relationships:

- May link to a `DispatchPing` through `sourcePingId`.
- May be assigned to multiple members via `assignedMemberIds`.
- May have `DispatchAssignment` records through `queueItemId`.
- May create timeline events through `queueItemId`.
- Always carries a `DispatchLinkedContext`.

Lifecycle:

- Created from pings requiring action, assist requests, manual assignments, or mock/local data.
- Can move from `new` to `pending_response`, `assigned`, `in_progress`, `blocked`, `escalated`, `needs_review`, `resolved`, or `cancelled`.
- Resolved/cancelled items are retained and sorted behind active work.

Persistence source:

- Stored locally in `DispatchPersistenceSnapshot.queueItems`.

Realtime behavior:

- Published as `queue_item_upsert`.
- Incoming updates are conflict-aware and merged by ID/idempotency key.

Offline behavior:

- Queued/failed/retrying local queue items replay through `replayQueuedDispatchActions`.
- Replay can mark delivery `retrying`, `recovered`, or `failed`.

Privacy considerations:

- Linked context may include location or operational details.
- Assignment and context actions must respect permissions.

## 4. DispatchAssignment

Purpose: Associates one team member with one queue item.

Required fields:

- `id`
- `queueItemId`
- `assigneeMemberId`
- `status`: `DispatchAssignmentStatus`
- `assignedAt`

Optional fields:

- `idempotencyKey`
- `version`
- `updatedAt`
- `acceptedAt`
- `completedAt`
- `notes`
- conflict fields

Status values:

- `unassigned`
- `offered`
- `accepted`
- `in_progress`
- `blocked`
- `completed`
- `declined`

Relationships:

- `queueItemId` references `DispatchQueueItem.id`.
- `assigneeMemberId` references `DispatchTeamMember.id`.
- Assignment changes can create `DispatchTimelineEvent` records.

Lifecycle:

- Created by queue assignment actions.
- Can move through offered, accepted, in progress, blocked, completed, or declined.
- Conflict handling keeps latest valid assignment state and marks sync changes when needed.

Persistence source:

- Stored locally in `DispatchPersistenceSnapshot.assignments`.

Realtime behavior:

- Published as `assignment_upsert`.
- Incoming assignments merge by ID/idempotency key.

Offline behavior:

- Local assignments replay only when their related queue item replay succeeds.

Privacy considerations:

- Assignment visibility should respect roster and queue permissions.
- Assignment does not contain private contact details.

## 5. DispatchAssistRequest

Purpose: Structured support/recovery request. In the current UI, assist submission creates a ping, queue item, and timeline event. The typed `DispatchAssistRequest` exists for adapter/schema readiness.

Required fields:

- `id`
- `assistType`: `DispatchAssistRequestType`
- `priority`
- `status`: `DispatchQueueItemStatus`
- `createdAt`
- `createdByMemberId`
- `targetMemberIds`
- `message`
- `requireAcknowledgment`
- `escalationState`

Optional fields:

- `idempotencyKey`
- `version`
- `updatedAt`
- `linkedContext`
- `sourcePingId`
- `queueItemId`
- conflict fields

Assist types:

- `vehicle`
- `medical`
- `navigation`
- `fuel`
- `water`
- `mechanical`
- `comms`
- `recovery`
- `general_support`

Relationships:

- May link to `DispatchPing` through `sourcePingId`.
- May link to `DispatchQueueItem` through `queueItemId`.
- Targets team members by ID.
- May carry linked context.

Lifecycle:

- Created by assist request composer as a product flow.
- Critical assist requests create emergency-style Dispatch coordination records.
- Can be represented in queue/timeline even when no standalone assist request persistence path exists.

Persistence source:

- No standalone assist request array exists in `DispatchPersistenceSnapshot`.
- Current durable representation is ping + queue item + timeline event.

Realtime behavior:

- No standalone realtime event type exists for assist requests.
- Assist behavior is broadcast via `ping_upsert`, `queue_item_upsert`, and `timeline_event_added`.

Offline behavior:

- Offline assist records replay through their ping, queue item, and timeline records.

Privacy/safety considerations:

- Assist is ECS team coordination only.
- It does not contact emergency services, SMS, email, or phone systems.

## 6. DispatchAcknowledgment

Purpose: Typed acknowledgment/receipt record for a member response to a ping or queue item. Current product flows primarily store acknowledgment through ping arrays and check-in responses; this entity exists for contract completeness and future persistence.

Required fields:

- `id`
- `pingId`
- `memberId`
- `status`: `acknowledged`, `accepted`, or `declined`
- `acknowledgedAt`

Optional fields:

- `idempotencyKey`
- `version`
- `queueItemId`
- `message`

Relationships:

- `pingId` references `DispatchPing.id`.
- `queueItemId` can reference `DispatchQueueItem.id`.
- `memberId` references `DispatchTeamMember.id`.

Lifecycle:

- Created when a recipient acknowledges, accepts, or declines.
- Check-in flows currently update `DispatchPing.acknowledgedByMemberIds` and `DispatchPing.checkInResponses`.

Persistence source:

- No standalone acknowledgment store exists in `DispatchPersistenceSnapshot`.
- Current durable representation is embedded in `DispatchPing`.

Realtime behavior:

- No standalone acknowledgment realtime event type exists.
- Acknowledgment state is propagated through `ping_upsert` and related queue/timeline updates.

Offline behavior:

- Offline acknowledgment is represented as an updated ping/queue/timeline record.

Privacy considerations:

- Should not expose private contact details.
- Message content should remain operational and privacy-safe.

- `id`
- `type`: `DispatchLinkedContextType`
- `title`

Optional fields:

- `subtitle`
- `coordinates`
- `routeSegmentId`
- `metadata`

Context types:

- `expedition`
- `pin`
- `waypoint`
- `route_segment`
- `resource`
- `vehicle`
- `power`
- `manual`

Relationships:

- Used by pings, queue items, assist requests, and timeline events.
- Produced by `dispatchContextAdapter` from mock data and safe existing pin/route/resource/vehicle/power sources.

Lifecycle:

- Selected in composers or assigned as fallback expedition context.
- Queue and ping cards display context type/title.
- Context actions can seed pings.

Persistence source:

- Embedded inside Dispatch records.

Realtime behavior:

Offline behavior:

- Embedded context remains available while offline.

Privacy considerations:

- `coordinates` may expose member or route location.
- Rendering and smart suggestions must respect location permissions.

## 9. DispatchDeliveryState

Purpose: Shared delivery/status vocabulary across pings, queue items, timeline events, and team member sync state.

Required use:

- `DispatchPing.status`
- `DispatchQueueItem.deliveryState`
- `DispatchTeamMember.syncState`
- Optional `DispatchTimelineEvent.deliveryState`

Lifecycle and offline behavior are described in the Shared Enums section.

Realtime behavior:

- Included in realtime payloads as part of the entity state.
- Realtime merge uses delivery state when determining stronger/safer state.

Privacy considerations:

- Delivery state does not expose private details by itself.

## 10. DispatchEscalationState

Purpose: Represents escalation ladder and recovery state for pings, queue items, assist requests, and timeline entries.

Required use:

- `DispatchPing.escalationState`
- `DispatchQueueItem.escalationState`
- `DispatchAssistRequest.escalationState`
- Optional `DispatchTimelineEvent.escalationState`

Lifecycle:

- Starts as `none`, `monitor`, or `recommended`.
- Manual escalation can advance ladder states.
- Terminal/recovery states include `resolved` and `recovered`.

Realtime/offline behavior:

- Escalation state is merged conflict-aware.
- Critical/emergency escalation is not silently overwritten by older non-escalated state.

Privacy/safety considerations:

- Emergency escalation remains ECS team coordination only.
- No emergency services are contacted.

## 11. DispatchQueuedOfflineAction

Purpose: Typed representation of a queued offline Dispatch action.

Required fields:

- `id`
- `idempotencyKey`
- `entityType`
- `actionType`
- `createdAt`
- `status`: `queued`, `replayed`, or `failed`

Optional fields:

- `replayedAt`
- `sourceEntityId`

Entity types:

- `ping`
- `queue_item`
- `assignment`
- `assist_request`
- `acknowledgment`
- `timeline_event`

Current implementation note:

- Although this type exists in `lib/dispatchTypes.ts`, the current replay implementation does not persist a separate array of `DispatchQueuedOfflineAction`.
- `lib/dispatchOfflineReplayAdapter.ts` currently replays queued local Dispatch entity records from `DispatchPersistenceSnapshot`: pings, queue items, assignments related to replayed queue items, and timeline events.

Lifecycle:

- Conceptually: `queued` -> `replayed` or `failed`.
- Currently represented by entity delivery/reliability states such as `queued`, `retrying`, `recovered`, and `failed`.

Realtime behavior:

- Replayed entities publish their normal realtime event types.

Offline behavior:

- Replay only runs for local records whose IDs start with `local-`.
- Successful replay updates entity delivery state.
- Failed replay preserves failed state for retry.

Privacy considerations:

- Offline action records should not store restricted contact/location details beyond the source entity reference.

## Persistence Snapshot

`DispatchPersistenceSnapshot` includes:

- `version`
- `expeditionId`
- `pings`
- `queueItems`
- `assignments`
- `timelineEvents`
- `updatedAt`

It does not currently include standalone arrays for:

- assist requests
- acknowledgments
- queued offline actions
- notifications

Those are represented through the durable entities above or policy helpers.

## Realtime Event Contract

Realtime event draft types:

- `{ type: 'ping_upsert'; ping }`
- `{ type: 'queue_item_upsert'; queueItem }`
- `{ type: 'assignment_upsert'; assignment }`
- `{ type: 'team_member_upsert'; teamMember }`
- `{ type: 'timeline_event_added'; timelineEvent }`

Realtime envelope adds:

- `id`
- `expeditionId`
- `originClientId`
- `occurredAt`

Events are ignored if they are for another expedition, from the same client, malformed, or already seen.

## Privacy And Safety Requirements

Dispatch data must preserve these constraints:

- Do not expose member coordinates unless `view_member_location` is allowed.
- Do not expose contact methods unless `view_member_contact` is allowed.
- Do not store or render raw technical IDs in user-facing copy where a label exists.
- Do not trigger SMS, email, phone, push, or emergency services from Dispatch data.
- Assist Request and Emergency Ping are ECS team coordination only.
- Notifications must be deduped, permission-aware, and disabled by default until verified.

## Future Schema Considerations

If/when Dispatch moves from local-first persistence to live backend tables, schema should preserve:

- expedition-scoped stable IDs
- idempotency keys
- version or updated-at conflict resolution
- actor/current user references
- linked context copies
- standalone acknowledgment and assist request tables if needed
- offline action replay metadata if a separate action queue is introduced
- privacy-safe audit descriptions
- notification dedupe keys
